# LIM-0001: macOS Preview Color Shift at Dissolve/Opacity Boundaries

## Status
- State: `open`
- Priority: `medium`
- Type: preview pipeline limitation (not data corruption)

## Summary
On macOS, users may observe a short color jump when playback enters or exits dissolve/fade/opacity transitions.
The artifact is usually 1 frame, sometimes 2 frames, and is most visible on high-contrast content.
This is a known behavior in the current preview pipeline.

## Scope
- Platform: macOS preview path.
- Feature area: dissolve, fade-in, fade-out, opacity ramps, and alpha compositing.
- Timing: transition boundaries (start/end of opacity-affected segments).
- Stage: preview playback/monitoring.

## What Users Usually See
- A brief "flash" where color looks different.
- Perceived as sudden contrast, saturation, gamma, or HSLA overlay tint change.
- Appears only at transition boundaries, not continuously.
- Significantly reduced when preview remains in stable `BGRA`.

## Technical Cause
The preview path is optimized for performance and may use:
- `NV12` for fast YUV decode/display,
- `BGRA` when alpha compositing is required.

At dissolve/opacity boundaries, frames may move between these paths.
If conversion assumptions are not fully identical for boundary frames, a short visual mismatch can appear.
Typical mismatch sources include:
- color matrix handling (for example, BT.709 conversion details),
- range interpretation (limited vs full),
- transfer/compositing order differences between paths.

## Why This Is Not Fully Eliminated Yet
The robust fix is a fully unified color-managed compositing path for preview, or forcing full-time BGRA.
Both approaches are valid but increase runtime cost:
- higher memory bandwidth,
- more GPU/CPU pressure,
- potential reduction in real-time smoothness on weaker systems.

Because of this tradeoff, the behavior is currently documented as a known limitation.

## What Is Not Affected
- Project file integrity.
- Timeline edit structure.
- Media source files.
- Export output is not automatically incorrect because of this preview artifact.

## Recommended Workarounds
- For color-critical checking, use a stable BGRA preview configuration.
- Validate final output by checking exported media frames near transitions.
- Avoid judging final grade from a single live boundary frame during playback.

## ACP Reply (Short)
This is a known macOS preview limitation. At dissolve/opacity boundaries, Anica may briefly switch between NV12 and BGRA preview paths, which can cause a 1-2 frame color mismatch. Your project and media are safe. For color-critical review, use BGRA-stable preview and verify the final exported output.

## ACP Reply (Detailed)
What you are seeing is a known preview artifact on macOS at fade/dissolve boundaries.  
Anica uses a performance-oriented preview path, and boundary frames may pass between NV12 (fast YUV path) and BGRA (alpha-compositing path).  
That handoff can produce a short color mismatch if conversion settings differ slightly at the boundary frame.  
This does not mean your timeline is corrupted. For final validation, check the exported result and use a stable BGRA preview mode when color accuracy is critical.
