## 1) Unified Graph Model

MotionLoom uses one unified `<Graph>` form. Do not generate or recommend `scope` on `<Graph>`.

Use these top-level building blocks together as needed:

1. `<Background ... />` for canvas background.
2. `<Scene id="...">...</Scene>` for generated vector/text/image composition.
3. `<Clip id="..." ... />` for input video/image clips.
4. `<World id="...">...</World>` for GLB actors, cameras, retarget maps, and actions.
5. `<Tex>`, `<Pass>`, and `<Present>` for explicit resource routing and final output.

`apply="clip|graph"` remains available for timing/layer effect behavior; it is not a graph mode.

### Scene graph structure

`<Scene>` must contain a `<Timeline>` as its root child. The timeline contains `<Track>` elements, each with `<Sequence>` or `<Chain>` children, and visual nodes live inside `<Layer>` within each sequence.

```xml
<Scene id="scene0">
  <Timeline>
    <Track id="main" space="world" z="0">
      <Sequence from="0s" duration="3s" out="hold">
        <Layer>
          <Text value="hello" x="center" y="center" fontSize="48" color="#ffffff" />
        </Layer>
      </Sequence>
    </Track>
  </Timeline>
</Scene>
```

- `space="world"` for world coordinates (default).
- `space="screen"` for screen coordinates (no camera transform).
- `z` controls draw order (higher = later).
- `zDepth` controls depth sorting (negative = closer).

## 3) DSL Tags (7 Core Tags)

1. `<Graph ...>`
2. `<Input ... />`
3. `<Tex ... />`
4. `<Buffer ... />`
5. `<Pass ... />`
6. `<Output ... />`
7. `<Present ... />`

Scene graphs additionally support:
- `<Background ... />` for full-frame graph backgrounds.
- `<Scene id="...">...</Scene>` for named scene resources.
- `<Timeline ...>` for time-based sequencing.
- `<Track ...>` for layered tracks within a timeline.
- `<Sequence ...>` for timed clips on a track.
- `<Chain ...>` for sequential scheduling of clips without hand-coded offsets.
- `<Layer ...>` for visual node containers within a sequence.
- `<Group ...>...</Group>` for grouped layout transform and opacity.
- `<Character ...>...</Character>` for dense vector character/face/hair composition.
- `<Mask ...>...</Mask>` for clipping children to a circle, rect, or path mask.
- `<Camera ...>...</Camera>` for optional flat world pan/zoom/rotation.
- `<Text ... />` for generated text/subtitle layers with animatable opacity.
- `<Image ... />` and `<Svg ... />` for visual assets.
- `<Rect ... />`, `<Circle ... />`, `<Line ... />`, `<Polyline ... />`, `<Path ... />`, and `<Shadow ... />` for motion-graphics/UI composition.
- `<Text ... width="..." lineHeight="..." />` for wrapped UI/caption text.
- `<Repeat ...>...</Repeat>` for repeated element patterns.
- `<Defs ...>...</Defs>` for gradients, brushes, and reusable assets.

Scene-authoring note:
- For new scene text/asset/UI composition, prefer `<Scene id="scene0">...</Scene>` with `<Timeline>` inside, and `<Present from="scene0" />`.
- Direct scene nodes (without `<Timeline>`) remain accepted for simple graphs for backwards compatibility.
- `Scene` draws UI/motion graphics; `Pass` applies post-process to a texture resource. Use `Scene -> Tex -> Pass -> Present` when both are needed.
- Do not use legacy pass syntax like `kind="raster"` for this flow.

## 4) Explicit vs Implicit Parameters

Rule:
- Required fields must be explicit.
- Optional fields can be omitted and rely on defaults.
- `effect` is now required on every `<Pass ... />` (all 7 process categories use the same mental model).

### `<Graph>`
- Required:
  - `fps`
  - `size`
- Forbidden:
  - `scope` (removed; use unified graph tags instead)
- Optional:
  - `id`
  - `version`
  - `apply` (default: `clip`)
  - `duration` (default: `2s` / `2000ms`)

`apply` behavior:
- `apply="clip"`: full clip span (default)
- `apply="graph"`: effect active only within `duration`

### `<Pass>`
- Required:
  - `id`
  - `effect`
  - `in`
  - `out`
- Optional:
  - `kernel` (if omitted, runtime resolves via `effect_kernel_map.kv`)
  - `kind` (default: `compute`)
  - `role` (`effect | transition`)
  - `transition`
  - `transitionFallback`
  - `transitionEasing`
  - `transitionClips`
  - `params`
  - `iterate`
  - `pingpong`
  - `cache`
  - `blend`
  - `loadOp`
  - `storeOp`

`blur_sharpen_detail_gaussian.wgsl` pass effect values:
- `gaussian_5tap_h`
- `gaussian_5tap_v`
- `box`
- `unsharp`

`transition_core.wgsl` pass effect values:
- `fade_in`
- `fade_out`
- `dip`
- `dissolve`

Transition timing guidance:
- Keep Graph at default `apply="clip"` (or omit `apply`).
- Do not depend on `Graph.duration` for transition timing.
- Put timing in `params`, for example:
  - `startSec`
  - `durationSec`
  - or explicit `progress` expression
  - optional `easing` in `params`
- `fade_out` special default: if `startSec` is omitted, it auto-anchors to clip tail (`clipDuration - durationSec`).

### Other nodes
- `<Input>` required: `id`
- `<Tex>` required: `id`, `fmt`
- `<Buffer>` required: `id`, `elemType`
- `<Output>` required: `id`, `to`
- `<Present>` required: `from`
- `<Background>` optional: `id`, `color`
- `<Text>` required: `value`; optional: `x`, `y`, `width`, `fontSize`, `lineHeight`, `color`, `opacity`, `visibleChars`, `maxLines`, `fontFamily`, `fontPath`
- `<Character>` optional container: `id`, `x`, `y`, `rotation`, `scale`, `opacity`; optimized for dense vector children such as `Group`, `Mask`, `Path`, `Polyline`, `Line`, `Circle`, `Rect`, and `Text`.
- `<Mask>` optional container: `id`, `shape` (`circle | rect | path`), `x`, `y`, `width`, `height`, `radius`, `d`, `opacity`; clips its children. Use `shape="circle"` with `x/y/radius`, `shape="rect"` with `x/y/width/height/radius`, or `shape="path"` with `d`.
- `<Camera mode="2d">` optional container: `x`, `y`, `targetX`, `targetY`, `anchorX`, `anchorY`, `zoom`, `rotation`, `opacity`, `follow`, `viewport`, `worldBounds`; children use world coordinates. The camera maps its target point to `(anchorX, anchorY)` inside the viewport. `follow` tracks a child node by `id`; `worldBounds` uses `x,y,width,height` to clamp camera movement.
- `<Line>` required: `x1`, `y1`, `x2`, `y2`; optional: `width`, `stroke`/`color`, `opacity`, `lineCap`, `taperStart`, `taperEnd`
- `<Polyline>` required: `points`; optional: `stroke`/`color`, `strokeWidth`/`width`, `opacity`, `trimStart`, `trimEnd`, `lineCap`, `lineJoin`, `taperStart`, `taperEnd`
- `<Path>` required: `d`; optional: `fill`, `stroke`/`color`, `strokeWidth`/`width`, `opacity`, `trimStart`, `trimEnd`, `lineCap`, `lineJoin`, `taperStart`, `taperEnd`. Use `stroke="none"` for fill-only shapes.

## 5) Duration Format Guidance

Supported examples:
- `duration="2s"`
- `duration="2.35s"`
- `duration="2350ms"`

If omitted:
- defaults to `2s`.

## 6) Scene Rendering Profiles

- `SceneRenderProfile::Cpu` — CPU rasterization path.
- `SceneRenderProfile::Gpu` — GPU accelerated path (preferred on macOS with `NV12` preview).
- `SceneRenderProfile::GpuProRes` — GPU path with ProRes output.

Scene `zDepth` uses camera-space depth: **negative is closer, positive is farther**.

## 7) Process Catalog APIs

- `process_effects()` — Returns the full built-in effect catalog.
- `process_effect_for_id(id)` — Looks up one effect by its ID.
- `process_effects_for_category(category)` — Lists effects by category name.
- `kernel_source_by_name(kernel)` — Returns embedded WGSL source for a known kernel.
- `is_known_process_kernel(kernel)` — Checks whether a kernel name is built-in.

## 8) GLB Helpers

- `load_glb_metadata(path)` — Load lightweight GLB metadata (nodes, meshes, joints, materials).
- `load_glb_mesh_data(path)` — Load full mesh data for world rendering.
- `diagnose_world_glb_gpu_plan(mesh)` — Diagnostic report for GPU world path suitability.
- `diagnose_world_graph_actor_gpu_frame(graph, actor_id, frame, asset_root)` — Diagnose one actor at a specific frame.

## 9) Recommended Effect Authoring Process (for ACP Guidance)

When user asks "how to write this effect", answer with this order:

1. Pick the graph building blocks: Background, Clip, Scene, World, Tex, Pass, Present.
2. Pick one of 10 process categories for each Pass.
3. Start with minimal graph (`Input -> Tex -> Pass -> Present`).
4. Use one kernel first (single pass), then add params.
5. Run and inspect runtime status:
   - `supported/total passes`
6. If needed, add second pass or branching graph.
7. Only then add advanced fields (`iterate`, `pingpong`, `cache`, blend/load/store).

## ACP Reply (Short)
Use unified `<Graph>` without `scope`. Use top-level tags to describe what is being composed, and use the 10 process categories for effect type. Keep required fields explicit (`Graph: fps+size`, `Pass: id+effect+in+out`) and omit optional fields for a clean script. `kernel` may be omitted when effect mapping exists. Default behavior is `apply="clip"` with default `duration="2s"` when omitted.

For scene authoring, always wrap visual nodes in `<Timeline><Track><Sequence><Layer>` when generating new scenes, or use the backwards-compatible direct-child form for simple one-off scripts.
