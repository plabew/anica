# ML-0007: Scene Control, Rigging, Deformation, and Editor Keyframes

Source of truth:
- `anica/crates/motionloom/src/dsl.rs`
- `anica/crates/motionloom/src/scene/dsl.rs`
- `anica/crates/motionloom/src/scene/model.rs`
- `anica/crates/motionloom/src/scene/render.rs`
- `motionloom-example/core/scene/cs-000027/` through `motionloom-example/core/scene/cs-000037/`

Use this document when an ACP or LLM workflow needs After Effects / DaVinci-style scene control: keyframes, character rigs, IK/FK, puppet pins, mesh topology, 2.5D layers, masks, or editable path morphs.

Machine-readable UI property metadata lives in `scene-ui-schema.json`. Use that file as the source of truth for editor property panels, not prose tables.

## 1) Current Scene Control Feature Map

| Need | Use | Core example |
|---|---|---|
| UI keyframes | `<AnimationTarget>` + `<Key>` | `cs-000036`, `cs-000037` |
| Classic curve expressions | `curve("time:value:ease, ...")` | most scene examples |
| Puppet pin deformation | `<Puppet>` + `<Pin>` | `cs-000034` |
| Manual puppet topology | `<MeshTopology>` + `<Vertex>` + `<Triangle>` | `cs-000035` |
| FK character controls | parented `<Group>` rotations or `<Action><Pose><Bone>` | `cs-000033` |
| IK character controls | `<Action><IK ... />` | `cs-000033` |
| Bone-driven artwork | `<Character rig="...">` + `<Part attachTo="bone_id">` | `cs-000033`, `cs-000037` |
| Raster character source | `<Character src="...">` | character/image tests and scene renderer |
| Editable path shape keys | `AnimationTarget property="d"` | `cs-000037` |
| 2.5D card/layer motion | `<Layer3D>` | `cs-000030`, `cs-000031` |
| Follow/reveal masks | `<Mask follow="node:id">` + group mask | `cs-000029` |
| Procedural face/jaw path | `<FaceJaw>` | `cs-000028` |

## 2) AnimationTarget + Key

`AnimationTarget` is the editor-friendly keyframe API. It is a direct child of `<Graph>` and targets a node by `id`.

Supported `property` values:
- `x`
- `y`
- `rotation`
- `scale`
- `scaleX`
- `scaleY`
- `skewX`
- `skewY`
- `transformOriginX`
- `transformOriginY`
- `opacity`
- `d`

`x/y/rotation/scale/scaleX/scaleY/skewX/skewY/transformOriginX/transformOriginY/opacity` values must be numeric strings. `d` values are SVG path strings. Each `Key` must use exactly one timing attribute: `time` or `frame`. Prefer `time="1.5s"` for UI-authored keys because it survives FPS changes; use `frame="45"` only when the key must stay frame-exact. Runtime converts both forms to seconds and writers preserve the original timing form. `Key.ease` uses the same easing names as `curve(...)`: `linear`, `ease_in`, `ease_out`, `ease_in_out`, or `ease(x1,y1,x2,y2)`.

```xml
<AnimationTarget node="card" property="rotation">
  <Key time="0s" value="0" ease="linear" />
  <Key time="0.5s" value="18" ease="ease_in_out" />
  <Key time="1s" value="0" ease="ease_out" />
</AnimationTarget>
```

Runtime behavior:
- `AnimationTarget` samples the value for the current frame before render.
- It overrides the targeted node property for that frame.
- It reuses the existing transform, opacity, path morph, puppet, and skeleton pipelines. It is not a separate renderer.

Use `AnimationTarget` when a future UI should let users drag a handle, rotate a bone, edit a path, or set editable keys. Keep `curve(...)` for compact hand-written procedural animation.

Editor data-model helpers:

- `extract_editable_animation_timeline(script)` reads `.motionloom` text into `EditableAnimationTimeline`.
- `upsert_editable_animation_target(script, target)` writes one node/property channel back to DSL.
- `replace_editable_animation_targets(script, targets)` replaces the full `AnimationTarget` set.

These helpers validate by parsing the generated script again, so UI write-back does not silently produce invalid DSL.

## 3) AnimationTarget Advanced Targets

### Puppet Pin x/y

For a `<Pin>`, `AnimationTarget property="x"` writes the animated target X, and `property="y"` writes the animated target Y. The pin's own `x/y` remain the rest/source anchor.

```xml
<Puppet id="cloth" x="0" y="0" width="240" height="150" density="high" amount="1">
  <Path d="M 0 0 L 240 0 L 240 150 L 0 150 Z" fill="#38bdf8" />
  <Pin id="cloth_anchor" x="20" y="24" fixed="true" radius="180" />
  <Pin id="cloth_pull" x="220" y="24" fixed="false" radius="180" strength="1" />
</Puppet>

<AnimationTarget node="cloth_pull" property="x">
  <Key time="0s" value="220" />
  <Key time="1s" value="245" ease="ease_in_out" />
</AnimationTarget>
<AnimationTarget node="cloth_pull" property="y">
  <Key time="0s" value="24" />
  <Key time="1s" value="115" ease="ease_in_out" />
</AnimationTarget>
```

Important coordinate rule: auto mesh coordinates run from `0..width` and `0..height`. Keep visual child geometry and pins in that local coordinate system, then move the whole puppet using `Puppet x/y`. Do not draw puppet children around negative coordinates unless you also provide matching manual topology.

### Path d

Use `AnimationTarget property="d"` for UI-edited path shape keys. Do not use `curve(...)` for `Path.d`.

```xml
<Path id="blob" d="M -60 0 C -30 -60 40 -60 60 0 C 30 60 -30 60 -60 0 Z" fill="#a78bfa" />
<AnimationTarget node="blob" property="d">
  <Key time="0s" value="M -60 0 C -30 -60 40 -60 60 0 C 30 60 -30 60 -60 0 Z" />
  <Key time="1s" value="M -90 -30 C -20 -80 80 -30 70 30 C 20 80 -80 40 -90 -30 Z" />
</AnimationTarget>
```

Path keys should keep compatible command structure when possible. Large structural differences can render but may not interpolate as expected.

### Bone rotation

Use `AnimationTarget node="bone_id" property="rotation"` to set a skeleton bone's rotation for editor/UI keyframes.

```xml
<Skeleton id="arm_rig">
  <Bone id="root" x="0" y="0" />
  <Bone id="arm" parent="root" x="0" y="0" rotation="0" />
</Skeleton>

<Character id="arm_character" rig="arm_rig" x="400" y="300">
  <Part id="arm_art" attachTo="arm">
    <Path d="M 0 -12 L 120 -12 L 120 12 L 0 12 Z" fill="#fb7185" />
  </Part>
</Character>

<AnimationTarget node="arm" property="rotation">
  <Key time="0s" value="-30" />
  <Key time="1s" value="30" ease="ease_in_out" />
</AnimationTarget>
```

If an `<ApplyAction>` also targets the same character, author carefully: action poses and IK can also affect the same bones. Prefer one driver per property for predictable UI editing.

## 4) Puppet + Pin

`Puppet` deforms child artwork through an auto mesh by default.

Key attributes:
- `width`, `height`: local mesh bounds.
- `density`: `low`, `medium`, `high`, `ultra`, or an explicit grid such as `7x7`.
- `amount`: deformation blend `0..1`.
- `x`, `y`, `rotation`, `scale`, `opacity`: normal transform properties.

`Pin` attributes:
- `x`, `y`: source/rest point in puppet local coordinates.
- `targetX`, `targetY`: animated destination. If omitted, target equals source.
- `fixed="true"`: lock target to source for anchors.
- `radius`: influence radius.
- `strength`: influence multiplier.
- `falloff`: `smooth`, `linear`, `gaussian`, or `none`.
- `vertex`: optional manual topology vertex id.

Use auto mesh for simple rectangular/image/cloth deformation. Use high density or larger radius to avoid overly coarse deformation.

## 5) MeshTopology For Advanced Puppet Control

`MeshTopology` is optional. Add it only when auto mesh cannot describe the desired deformation.

Supported child tags:
- `<Vertex id="..." x="..." y="..." />`
- `<Triangle a="v0" b="v1" c="v2" />`
- `<Edge a="v0" b="v1" />`
- `<Region id="..." vertices="v0,v1,v2" />`

Pins can attach to vertices:

```xml
<Puppet id="leaf" width="300" height="200" amount="1">
  <MeshTopology>
    <Vertex id="root" x="0" y="100" />
    <Vertex id="tip" x="300" y="100" />
    <Vertex id="top" x="150" y="0" />
    <Triangle a="root" b="tip" c="top" />
  </MeshTopology>
  <Pin id="root_pin" vertex="root" fixed="true" />
  <Pin id="tip_pin" vertex="tip" targetX="330" targetY="60" radius="220" />
  <Path d="M 0 100 C 90 0 210 0 300 100 C 210 200 90 200 0 100 Z" fill="#22c55e" />
</Puppet>
```

Do not force every example to write topology. The default should be `Puppet + Pin + auto mesh`; topology is for expert cases.

## 6) Skeleton, Action, FK, IK, Character, Part

Skeleton rig flow:

1. Define a top-level `<Skeleton id="...">`.
2. Define bones with `id`, optional `parent`, and local `x/y/rotation`.
3. Define an `<Action id="..." skeleton="...">` with `<Pose>` and optional `<IK>`.
4. Render using `<Character rig="skeleton_id">` and child `<Part attachTo="bone_id">`.
5. Apply the action with `<ApplyAction target="character_id" action="action_id" at="0s" />`.

Two-bone IK:

```xml
<IK root="shoulder" mid="elbow" end="wrist"
    targetX="260" targetY="-80" bend="1" weight="1" />
```

CCD chain IK:

```xml
<IK chain="finger_1,finger_2,finger_3,finger_tip"
    targetX="24" targetY="-120" iterations="10" weight="1" />
```

Rules:
- `root -> mid -> end` must be a direct parented two-bone chain.
- `chain` ids must be direct parent-child bones; the last id is the end effector.
- `targetX/targetY`, `bend`, `weight`, and `iterations` can be numeric expressions or `curve(...)` values.
- FK is still valid: use explicit parent-child `<Group>` rotations or pose bone rotations when exact manual arcs are preferred.
- Do not show FK and IK controls simultaneously in user-facing examples unless the purpose is comparison. Phase them with opacity or split scenes.

## 7) Character Raster Source

`Character` can draw vector children and/or a raster source image.

```xml
<Character id="person" src="assets/sample_assets/characters/character1/IMAGECHARACTER.png" x="400" y="300" scale="1" />
```

Aliases for `src`: `image` and `path`.

Supported raster source forms are the same as `<Image>`:
- project/asset-relative PNG/JPG paths resolved by the host asset resolver;
- `data:image/png;base64,...` or `data:image/jpeg;base64,...` for self-contained examples.

Raster SVG data URIs belong in `<Svg>`, not raster `Character src`.

## 8) Layer3D, Mask Follow, and FaceJaw

### Scene Camera and Layer3D UI Schema

Use `scene-ui-schema.json` when building UI property panels. It currently defines:

- `SceneCamera`: display label `Scene Camera`, DSL tag `<Camera />`, camera-track-only placement, and editable groups for Transform, Anchor, Follow, Bounds, and Shake.
- `Layer3D`: display label `Layer3D / 2.5D Layer`, DSL tag `<Layer3D>`, normal scene-layer placement, and editable groups for Transform and Projection.

Do not expose `Camera2.5D` as a component. The 2.5D object is `Layer3D`; the camera remains `Scene Camera`.

### Layer3D

`Layer3D` is AE-like 2.5D for flat scene content. Use it for cards, panels, logos, and UI surfaces that need depth and tilt.

Typical attributes: `x`, `y`, `z`, `rotationX`, `rotationY`, `rotationZ`, `scale`, `opacity`, `perspective`.

Use `zDepth`/`z` carefully: negative depth is closer in scene depth sorting, while track `z` remains draw-order tie-break.

Do not treat `Layer3D` as a camera component. Scene camera motion stays in `<Track role="camera"><Sequence><Camera ... /></Sequence></Track>`. `Layer3D` is the flat 2.5D object/layer transform that the camera can view.

### Mask follow

Use `Mask follow="node:id"` when a reveal/matte needs to track a moving scene node. Apply it to a `Group` with `mask="mask_id"` or `maskFrom` depending on the matte pattern.

### FaceJaw

`FaceJaw` generates a face/jaw curve from simple parameters. Use it when a character face outline should be adjustable without hand-editing a full path. Animate parameters such as `jawEase`, `chinSharpness`, or trim values with `curve(...)`.

## 9) Generator Rules For LLMs

- Prefer `curve(...)` for compact hand-authored numeric animation.
- Prefer `AnimationTarget + Key time="..."` when the result should map to a future UI keyframe editor.
- Never animate `Path.d` with `curve(...)`; use `AnimationTarget property="d"`.
- For Pin UI editing, animate the pin target through `AnimationTarget property="x/y"`; keep source `Pin x/y` as rest coordinates.
- Keep puppet child geometry in the same local coordinate system as the mesh.
- Use `MeshTopology` only when auto mesh is insufficient.
- Use one driver per property when possible: avoid mixing `AnimationTarget`, `curve(...)`, `<Action>`, and `<IK>` on the same exact property unless the layering is intentional.
- Keep examples parseable and renderable; validate with `cargo run -p motionloom --example render_file_frame -- <script> /tmp/frame.png 0 cpu`.

## ACP Short Reply

Use `AnimationTarget + Key time="..."` for editor-style keys, `Puppet + Pin` for AE-style pin deformation, optional `MeshTopology` for manual triangles, and `Skeleton/Action/IK/Character/Part` for rigs. Keep `curve(...)` as the compact expression API, but use `AnimationTarget property="d"` for path shape keys and `AnimationTarget node="pin_id" property="x/y"` for draggable pin targets.
