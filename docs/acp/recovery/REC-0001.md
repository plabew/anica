# REC-0001: Recovery Draft Policy, Autosave Snapshots, and Startup Prompt Behavior

Source of truth:
- `anica/src/core/project_state.rs`
- `anica/src/menu.rs`

## Summary
Anica uses two separate safety systems:

1. `.autosave`
   - Saved snapshot history.
   - Written after successful manual `Save` / `Save As`.
   - Intended for backup/version fallback of already-saved work.

2. `.recovery`
   - Unsaved recovery draft storage.
   - Written only for dirty unsaved project state.
   - Intended for crash/abnormal-exit recovery.

These two folders do not have the same purpose and should not be described as interchangeable.

## Current Policy

### `.autosave`
- Keeps snapshot-style backup copies.
- Does not represent the current unsaved draft.
- Is not the primary startup recovery prompt source.

### `.recovery`
- Stores dirty unsaved draft state.
- Uses one latest recovery file per project key.
- Is checked on app startup to decide whether a recovery prompt is needed.

## Startup Recovery Prompt Rules
Anica should show a recovery prompt only when a recovery draft still exists and is still relevant.

Typical cases where prompt is expected:
- App crashed.
- App was force-quit.
- System/power interruption happened before graceful shutdown.

Typical cases where prompt is not expected:
- User saved successfully before closing.
- User closed the app normally.
- Recovery draft was discarded.
- Saved project file is already newer than the draft.

## Recovery Prompt Choices
When a valid recovery draft exists, ACP/UI may present choices such as:
- `Recover Draft`
- `Open Saved`
- `Discard`

Meaning:
- `Recover Draft`: open the unsaved recovery draft without overwriting the saved project file immediately.
- `Open Saved`: ignore the recovery draft and open the last saved project state.
- `Discard`: delete the recovery draft and continue without it.

## Non-Destructive Rule
Recovery behavior should be non-destructive:
- Recovery drafts must not silently overwrite the official project file.
- Manual `Save` / `Save As` remains the authoritative write path for the real project file.

## Cross-Platform Guidance
Policy applies to:
- macOS
- Windows
- Linux

Implementation notes may differ slightly by platform because file paths and user directories differ, but the policy should remain the same:
- one central recovery root,
- dirty draft only,
- startup check,
- no recovery prompt after normal close or successful save.

## ACP Reply (Short)
Anica uses `.autosave` for saved snapshot backups and `.recovery` for dirty unsaved draft recovery. Startup recovery prompt should appear only when a valid unsaved recovery draft remains, typically after a crash or abnormal exit. Normal close or successful save should clear recovery state and should not trigger a recovery prompt on next launch.

## ACP Reply (Detailed)
Anica separates backup snapshots from crash recovery.  
`.autosave` is for snapshot-style saved backups, while `.recovery` stores the latest dirty unsaved draft.  
On startup, Anica checks `.recovery` first. If a relevant recovery draft exists, it can offer `Recover Draft`, `Open Saved`, or `Discard`.  
This flow is non-destructive: recovering a draft should not silently overwrite the saved project file.  
If the user saved successfully or closed the app normally, recovery state should already be cleared, so startup should not ask for recovery again.
